# Phase 2.5: Translator Consolidation - Context

**Gathered:** 2026-03-14
**Status:** Ready for planning
**Source:** Session discussion (inserted phase)

<domain>
## Phase Boundary

Consolidate translator functions into config class methods using Pydantic v2 features. This refactor is required before Entry Point Discovery to establish a consistent plugin interface where config classes provide `to_adbc_kwargs()` instead of separate translator functions.

</domain>

<decisions>
## Implementation Decisions

### Translator Interface
- All 12 built-in config classes get `to_adbc_kwargs()` method
- Method returns `dict[str, str]` suitable for ADBC connection
- Uses Pydantic v2 features:
  - `Field(alias="...")` for ADBC key prefixes (e.g., `adbc.snowflake.sql.account`)
  - `@field_serializer` for transformations (bool→'true'/'false', SecretStr→string)
  - `@model_validator` for complex logic (Databricks URI construction)
- Base call: `self.model_dump(exclude_none=True, by_alias=True)`

### Registry Changes
- Registry stores only `(config_class, driver_path)` — translator removed
- `get_translator()` becomes `config.to_adbc_kwargs()` call
- `register_backend()` signature changes: `register_backend(name, config_class, driver_path)`
- Built-in backends use same lazy registration pattern

### Backend-by-Backend Migration
- Snowflake (115 lines): Most complex — aliases for `adbc.snowflake.sql.*` keys, serializers for bools and SecretStr
- DuckDB (27 lines): Simplest — just `path` and optional `access_mode`
- Databricks (60 lines): URI construction via `@model_validator`, two modes handled
- BigQuery, PostgreSQL, MySQL, MSSQL, Trino, FlightSQL, Redshift, ClickHouse, SQLite, Firebolt: Follow same pattern

### Test Strategy
- All existing translator tests must pass unchanged
- Unit tests call `config.to_adbc_kwargs()` instead of `translate_<backend>(config)`
- Integration tests verify pool creation still works
- Phase 1 semi-integration tests (real driver imports) unchanged

### Claude's Discretion
- Exact field alias names (follow ADBC driver conventions)
- Whether to use `@field_serializer` per field or model-wide serializer
- Import organization for serializers
- Whether to create a `BaseWarehouseConfig.to_adbc_kwargs()` base implementation

</decisions>

<specifics>
## Specific Ideas

- "One config class, one method" — translators embedded in configs, no separate files
- "model_dump is the transformer" — leverage Pydantic's built-in serialization rather than manual dict construction
- "All backends follow the same pattern" — consistency across 12 implementations

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `_translators.py`: Central dispatch, will be simplified to call config method
- `_<backend>_translator.py`: 12 individual translator files,will be deleted or converted
- `_base_config.py`: `WarehouseConfig` Protocol — needs `to_adbc_kwargs()` method added
- `_<backend>_config.py`: 12 config classes — need `to_adbc_kwargs()` implementation

### Established Patterns
- Translators return `dict[str, str]` for ADBC dbapi.connect()
- None values omitted from output
- Booleans serialized as lowercase strings
- SecretStr values extracted via `.get_secret_value()`
- ADBC key prefixes added manually (e.g., `adbc.snowflake.sql.*`)

### Integration Points
- `_translators.py:translate_config()` → calls `config.to_adbc_kwargs()`
- `_registry.py:get_translator()` → removed (translator built into config)
- `_pool_factory.py:create_pool()` → calls `translate_config(config)` unchanged
- Tests: `test_translators.py` → tests config method instead of function

### Files Modified
- `_base_config.py`: Add `to_adbc_kwargs()` to Protocol
- `_<backend>_config.py` (12 files): Add `to_adbc_kwargs()` implementation, add aliases/serializers
- `_translators.py`: Simplify to single method call
- `_registry.py`: Remove translator from registration tuple, simplify lookup
- `tests/test_translators.py`: Update to test config method

</code_context>
