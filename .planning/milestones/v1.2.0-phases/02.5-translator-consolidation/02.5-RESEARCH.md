# Phase 2.5: Translator Consolidation - Research

**Researched:** 2026-03-14
**Domain:** Pydantic v2 model serialization, ADBC driver configuration
**Confidence:** HIGH

## Summary

This phase consolidates 12 separate translator functions into method implementations on config classes using Pydantic v2's built-in serialization capabilities. The current architecture uses manual dict construction in `_xxx_translator.py` files; the target architecture embeds translation logic in `to_adbc_kwargs()` methods on config classes using `Field(alias=...)` and `@field_serializer`.

The primary challenge is that boolean defaults must always be emitted ('true'/'false'), and URI-construction backends require post-validation logic. Pydantic v2's `model_dump(exclude_none=True, by_alias=True)` handles most fields, but custom serializers are needed for:
1. Boolean fields → lowercase string values
2. SecretStr fields → `get_secret_value()` extraction
3. URI construction from composite fields (Databricks, PostgreSQL, MySQL, Redshift)

**Primary recommendation:** Implement `to_adbc_kwargs()` using `model_dump(exclude_none=True, by_alias=True)` as base, with `@field_serializer` decorators for booleans and SecretStr, and `@model_validator(mode='after')` override for URI-construction backends.

### Pattern 1: Base Implementation (Simple Backend)

**What:** Config classes use `model_dump(exclude_none=True, by_alias=True)` with Field aliases for ADBC-prefixed keys.

**When to use:** Backends with simple key mapping (BigQuery, SQLite, ClickHouse).

**Example (BigQuery):**
```python
from pydantic import Field

class BigQueryConfig(BaseWarehouseConfig):
    auth_type: str | None = Field(default=None, alias="adbc.bigquery.sql.auth_type")
    project_id: str | None = Field(default=None, alias="adbc.bigquery.sql.project_id")
    # ...

    def to_adbc_kwargs(self) -> dict[str, str]:
        """Convert config to ADBC driver kwargs."""
        # SecretStr fields need custom serialization
        result = self.model_dump(exclude_none=True, by_alias=True)
        # Handle SecretStr fields that model_dump would return as SecretStr objects
        # ... see Pattern 3 for SecretStr handling
        return result
```

**Source:** Pydantic v2 documentation on `model_dump()` and Field aliases.

### Pattern 2: Field Serializers for Booleans

**What:** Use `@field_serializer` decorator to convert boolean fields to 'true'/'false' strings.

**When to use:** All backends with boolean fields (Snowflake, FlightSQL, MSSQL, Trino).

**Example:**
```python
from pydantic import field_serializer

class SnowflakeConfig(BaseWarehouseConfig):
    tls_skip_verify: bool = False
    ocsp_fail_open_mode: bool = True

    @field_serializer("tls_skip_verify", "ocsp_fail_open_mode", mode="plain")
    def serialize_bool(self, value: bool) -> str:
        """Serialize boolean to 'true'/'false' string for ADBC."""
        return str(value).lower()

    def to_adbc_kwargs(self) -> dict[str, str]:
        return self.model_dump(exclude_none=True, by_alias=True)
```

**Important:** Boolean fields with defaults are always included in `model_dump()` output. `exclude_none=True` does NOT exclude booleans.

**Source:** Pydantic v2 documentation on `@field_serializer`.

### Pattern 3: Field Serializers for SecretStr

**What:** Use `@field_serializer` to extract `SecretStr` values.

**When to use:** All backends with SecretStr fields (Snowflake, BigQuery, FlightSQL, Databricks, etc.).

**Example:**
```python
from pydantic import SecretStr, field_serializer

class SnowflakeConfig(BaseWarehouseConfig):
    password: SecretStr | None = None
    private_key_pem: SecretStr | None = None

    @field_serializer("password", "private_key_pem", mode="plain", when_used="json")
    def serialize_secret(self, value: SecretStr | None) -> str | None:
        """Extract SecretStr value for ADBC kwargs."""
        if value is None:
            return None
        return value.get_secret_value()

    def to_adbc_kwargs(self) -> dict[str, str]:
        # model_dump will call the serializer
        return self.model_dump(exclude_none=True, by_alias=True, mode="json")
```

**Note:** The `mode="json"` parameter in `model_dump()` triggers JSON serialization mode, which invokes serializers. Without it, `SecretStr` objects stay as `SecretStr`.

**Source:** Pydantic v2 documentation on serialization modes.

### Pattern 4: Model Validator for URI Construction

**What:** Use `@model_validator(mode='after')` to build URI from composite fields.

**When to use:** Databricks, PostgreSQL, MySQL, Redshift (backends that construct URIs).

**Example (Databricks):**
```python
from pydantic import model_validator

class DatabricksConfig(BaseWarehouseConfig):
    uri: SecretStr | None = None
    host: str | None = None
    http_path: str | None = None
    token: SecretStr | None = None

    @model_validator(mode="after")
    def build_adbc_uri(self) -> Self:
        """Build ADBC URI from decomposed fields if uri not provided."""
        # This validator runs AFTER field validation
        # The to_adbc_kwargs() method will handle URI-first vs decomposed mode
        return self

    def to_adbc_kwargs(self) -> dict[str, str]:
        if self.uri is not None:
            return {"uri": self.uri.get_secret_value()}
        # Decomposed mode - build URI
        from urllib.parse import quote
        assert self.host is not None
        assert self.http_path is not None
        assert self.token is not None
        encoded = quote(self.token.get_secret_value(), safe="")
        return {"uri": f"databricks://token:{encoded}@{self.host}:443{self.http_path}"}
```

**Source:** Current `_databricks_translator.py` pattern, adapted to method.

### Pattern 5: Alias for Python Keyword Conflicts

**What:** Use `validation_alias` for Python keyword conflicts (e.g., `schema`).

**When to use:** Snowflake and Databricks config classes (already in use).

**Example (already implemented):**
```python
from pydantic import Field

class SnowflakeConfig(BaseWarehouseConfig):
    schema_: str | None = Field(default=None, validation_alias="schema", alias="schema")
```

**Note:** The field is named `schema_` internally but can be passed as `schema` and serializes as `schema`. For `model_dump(by_alias=True)`, this outputs `"schema": value` not `"schema_": value`. For ADBC, use `alias="adbc.snowflake.sql.schema"`.

**CRITICAL:** For ADBC output, use `alias="adbc.snowflake.sql.schema"` not just `alias="schema"`.

### Anti-Patterns to Avoid

- **Don't use `model_dump()` without `mode="json"` for SecretStr**: Returns `SecretStr` objects instead of strings
- **Don't rely on `exclude_none=True` to exclude booleans**: Booleans with defaults are always included
- **Don't try to serialize Path objects directly**: Use `str(config.private_key_path)` in `to_adbc_kwargs()` override
- **Don't forget to handle `exclude_none=True` for optional fields**: Pydantic preserves None values by default

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Key name mapping | Manual dict with key-by-key construction | `Field(alias="adbc.xxx.yyy")` + `model_dump(by_alias=True)` | Pydantic handles mapping; less error-prone |
| Boolean serialization | `str(config.bool_field).lower()` in method | `@field_serializer("field_name", mode="plain")` | Centralized, reusable, testable |
| SecretStr extraction | `if field: field.get_secret_value()` everywhere | `@field_serializer` with `mode="json"` | Automatic during serialization |
| None exclusion | Manual `if field is not None:` blocks | `model_dump(exclude_none=True)` | Pydantic handles conditionals |

**Key insight:** Pydantic v2's serialization system is designed for exactly this use case — transforming model fields to external formats. Leverage it fully rather than rebuilding dict construction manually.

## Common Pitfalls

### Pitfall 1: SecretStr Not Extracted in model_dump()

**What goes wrong:** `model_dump()` returns `SecretStr` objects, not strings.

**Why it happens:** By default, Pydantic preserves field types. `SecretStr` is designed to hide values in repr/str.

**How to avoid:** Use `model_dump(mode="json")` with `@field_serializer(when_used="json")` or call `get_secret_value()` manually in `to_adbc_kwargs()`.

**Warning signs:** Test assertions comparing `SecretStr(...)` instead of plain string values.

### Pitfall 2: Boolean Defaults Not Emitted by model_dump()

**What goes wrong:** Expecting `exclude_none=True` to exclude booleans, but booleans are not None.

**Why it happens:** `exclude_none=True` only excludes fields where the value IS None. Boolean `False` is not None.

**How to avoid:** Boolean fields with defaults are always included. Ensure `@field_serializer` handles them.

**Warning signs:** Tests failing with missing boolean keys in output dict.

### Pitfall 3: Field Alias vs Validation Alias Confusion

**What goes wrong:** Using `validation_alias` when you need `alias` for serialization, or vice versa.

**Why it happens:** Pydantic v2 separates input (validation_alias) from output (alias).

**How to avoid:**
- `validation_alias` — used when deserializing input (e.g., `{"schema": "PUBLIC"}` → `schema_` field)
- `alias` — used when serializing output (e.g., `schema_` field → `{"adbc.snowflake.sql.schema": "PUBLIC"}`)
- Use `Field(alias="output_key", validation_alias="input_key")` for different aliases

**Warning signs:** Tests failing because output keys don't match ADBC expected keys.

### Pitfall 4: URI Construction Backends Need Override

**What goes wrong:** Calling `model_dump()` on Databricks/PostgreSQL/MySQL/Redshift configs returns all fields, not just the computed URI.

**Why it happens:** These backends construct a single `uri` key from multiple fields; they don't emit individual fields.

**How to avoid:** Override `to_adbc_kwargs()` completely for URI-construction backends. Don't use `model_dump()` at all for these.

**Warning signs:** Tests expecting `{"uri": "..."}` receiving `{"host": ..., "port": ..., ...}`.

### Pitfall 5: Path Objects Need String Conversion

**What goes wrong:** `model_dump()` returns `Path` objects for `private_key_path` fields.

**Why it happens:** Pydantic serializes `Path` as `Path` by default.

**How to avoid:** Add `@field_serializer("private_key_path", mode="plain")` returning `str(value)`.

**Warning signs:** Type errors or ADBC driver not finding the path.

## Code Examples

### DuckDB Config (Simplest Case)

```python
# After migration - DuckDBConfig with to_adbc_kwargs
class DuckDBConfig(BaseWarehouseConfig):
    database: str = ":memory:"
    read_only: bool = False

    @field_serializer("read_only", mode="plain")
    def serialize_bool(self, value: bool) -> str:
        return str(value).lower()

    def to_adbc_kwargs(self) -> dict[str, str]:
        # DuckDB uses 'path' not 'database', and 'access_mode' for read_only
        result: dict[str, str] = {"path": self.database}
        if self.read_only:
            result["access_mode"] = "READ_ONLY"
        return result
```

### Snowflake Config (Most Complex Case)

```python
# After migration - SnowflakeConfig with to_adbc_kwargs
from pydantic import Field, SecretStr, field_serializer

class SnowflakeConfig(BaseWarehouseConfig):
    # Identity fields
    account: str
    user: str | None = None

    # Auth fields with SecretStr
    password: SecretStr | None = Field(None, alias="username")  # Note: 'username' not 'password' for ADBC# Wait, no - password maps to 'password', user maps to 'username'

    # Boolean fields (always included)
    tls_skip_verify: bool = Field(False, alias="adbc.snowflake.sql.client_option.tls_skip_verify")

    # ... many more fields ...
```

**Actually, reviewing current translator:** Snowflake uses explicit keys, not Field aliases. The keys like `adbc.snowflake.sql.account` are ADBC driver options. Using aliases would mean:
- `account` field → `adbc.snowflake.sql.account`
- `tls_skip_verify` field → `adbc.snowflake.sql.client_option.tls_skip_verify`

This is the correct approach for Pydantic-based serialization.

## Open Questions

1. **Base class `to_adbc_kwargs()` implementation?**
   - What we know: All configs need this method; some can use inherited logic
   - What's unclear: Whether to implement in `BaseWarehouseConfig` as abstract method or provide default implementation
   - Recommendation: Add abstract method to `WarehouseConfig` Protocol, provide default in `BaseWarehouseConfig` that raises `NotImplementedError`. Each backend overrides.

2. **Serializer registration strategy?**
   - What we know: `@field_serializer` can serialize single or multiple fields
   - What's unclear: Whether to use one serializer per field or one serializer for all bools/SecretStrs
   - Recommendation: One serializer for all boolean fields, one for all SecretStr fields. Cleaner than per-field decorators.

3. **Import organization for serializers?**
   - What we know: Need `from pydantic import Field, SecretStr, field_serializer, model_validator`
   - What's unclear: Whether to import in each config file or use a shared module
   - Recommendation: Import directly in each config file. Keeps imports minimal and explicit.
