Rewrite `src/adbc_poolhouse/_pool_factory.py`:

1. Add `overload` to the typing imports:
```python
from typing import TYPE_CHECKING, overload
```

2. Create `_create_pool_impl()` private function (before `create_pool`). This is the shared implementation that both `create_pool()` and `managed_pool()` call. Follow the exact pattern from RESEARCH.md "Pattern 1: Shared Implementation Helper":
   - Parameters: `config: WarehouseConfig | None`, `driver_path: str | None`, `db_kwargs: dict[str, str] | None`, `entrypoint: str | None`, `dbapi_module: str | None`, plus the 5 pool tuning params.
   - Dispatch: if config is not None -> extract from config methods. elif driver_path is not None -> use driver_path (require db_kwargs). elif dbapi_module is not None -> use dbapi_module (require db_kwargs, pass "" for driver_path). else -> raise TypeError.
   - Also check mutual exclusivity: if both driver_path and dbapi_module are not None, raise TypeError("create_pool() accepts driver_path or dbapi_module, not both").
   - After dispatch, call `create_adbc_connection()`, create QueuePool, attach _adbc_source, register reset listener, return pool.
   - The QueuePool creation, _adbc_source attachment, and event listener registration are moved from the current `create_pool()` into `_create_pool_impl()`.

3. Add three `@overload` stubs for `create_pool()` per RESEARCH.md "Pattern 2":
   - Overload 1: `config: WarehouseConfig` positional, keyword-only pool tuning, returns `QueuePool`
   - Overload 2: keyword-only `driver_path: str`, `db_kwargs: dict[str, str]`, `entrypoint: str | None = None`, pool tuning, returns `QueuePool`
   - Overload 3: keyword-only `dbapi_module: str`, `db_kwargs: dict[str, str]`, pool tuning, returns `QueuePool`
   - Implementation: all params with `None` defaults, calls `_create_pool_impl()`

4. Keep `close_pool()` unchanged.

5. Add three `@overload` stubs for `managed_pool()` per RESEARCH.md "Pattern 3":
   - Return type in overload stubs: `contextlib.AbstractContextManager[sqlalchemy.pool.QueuePool]`
   - Implementation uses `@contextlib.contextmanager` returning `Iterator[QueuePool]`
   - Implementation body calls `_create_pool_impl()` directly (not `create_pool()` -- avoids overload forwarding issues with basedpyright)
   - try/yield/finally/close_pool pattern stays the same

6. Keep `_release_arrow_allocators()` unchanged.

7. Add `import contextlib` to the `if TYPE_CHECKING` block (for `AbstractContextManager` in overload stubs). Actually, `contextlib` is already imported at module level -- just use `contextlib.AbstractContextManager` directly in the overload return types.

Important: The docstrings for `create_pool()` and `managed_pool()` will be updated in Plan 02 (documentation task). For now, update the implementation signature docstring to briefly mention the three call patterns, but defer the full Google-style docstring rewrite.
