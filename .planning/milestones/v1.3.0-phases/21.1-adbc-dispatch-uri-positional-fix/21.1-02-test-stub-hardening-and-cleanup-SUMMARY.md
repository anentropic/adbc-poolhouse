# Phase 21.1 Plan 02: Test stub hardening and cleanup Summary

The three driver-installed test paths that should have caught the Family A'
dispatch bug — `TestQuackImports`, `TestPostgreSQLImports`,
`TestFlightSQLImports` — now use signature-preserving stubs whose parameter
lists match the live `adbc_driver_*.dbapi.connect` shapes. A duplicate
`test_quack_returns_short_name` at module scope in `tests/test_drivers.py`
is removed, leaving the canonical `TestPyPIDriverPath` method as the single
attributable assertion.

## What changed

`tests/test_driver_imports.py` — for each of the three target classes, the
driver-installed branch was rewritten from:

```python
with patch("adbc_driver_quack.dbapi.connect", return_value=mock_conn) as mock_connect:
    pool = create_pool(config)
mock_connect.assert_called_once()
assert "uri" in mock_connect.call_args.kwargs
```

to a signature-preserving stub:

```python
captured: dict[str, Any] = {}

def stub_connect(uri, *, db_kwargs=None, conn_kwargs=None, autocommit=False):
    captured["uri"] = uri
    captured["db_kwargs"] = db_kwargs
    return mock_conn

with patch("adbc_driver_quack.dbapi.connect", new=stub_connect):
    pool = create_pool(config)

assert captured["uri"] == "quack://h"
assert "uri" not in (captured.get("db_kwargs") or {})
```

The stub reproduces the live driver's `connect()` shape — `uri` as
required-positional, `db_kwargs` as keyword (KEYWORD_ONLY for Quack,
POSITIONAL_OR_KEYWORD for Postgres / FlightSQL). Because the dispatcher in
`_driver_api.create_adbc_connection` uses `inspect.signature` to choose its
branch, a real function literal forces the Family A' route end-to-end.

`tests/test_drivers.py` — the module-scope function
`test_quack_returns_short_name` (and its docstring) at line 184 was deleted.
The canonical `TestPyPIDriverPath.test_quack_missing_returns_package_name`
at line 122 — same `find_spec → None` patch, same
`_driver_path() == "adbc_driver_quack"` assertion — remains in place.

## Why it matters

`MagicMock`'s wildcard `*args, **kwargs` signature was the load-bearing
reason the bug shipped for three releases. When `inspect.signature` reads
the patched `connect`, the bare mock looks like it accepts anything, so the
dispatcher's Family A' detection (required-positional `uri` + `db_kwargs`
present) never fires and the buggy default branch passes silently. Pinning
the patch target to a function literal with the affected shape closes that
gap; future signature-shape regressions of this class fail at the test
layer.

The duplicate removal addresses threat T-21.1-05 (repudiation / silent
pass): two identical assertions for the same behaviour give a false signal
of double coverage, so a future break of one would still appear green via
the other. Single source of truth — single failure signal.
