# Requirements: v1.2.0

**Milestone**: Plugin/Extensibility API  
**Defined**: 2026-03-10  
**Status**: PLANNED  

---

## Core Value

v1.2.0 enables 3rd party libraries and users to add custom backend providers without modifying adbc-poolhouse:
- Entry point auto-discovery for published plugin packages
- Manual registration API for in-project backends
- Clear documentation for plugin authors

---

## Quality Gates

All features must pass these gates before phase completion:

1. **Unit tests**: All new public APIs have unit tests
2. **Integration tests**: Entry point discovery tested with real dummy plugin
3. **Type checking**: `uv run basedpyright` passes with strict mode
4. **Linting**: `uv run ruff check` passes
5. **Formatting**: `uv run ruff format --check` passes
6. **Coverage**: New code covered by tests (no hard threshold, but must be reasonable)

## Registry Infrastructure

### REG-01: Backend Registry

Create an internal registry to store backend definitions.

**Acceptance:**
- `_BackendRegistry` class stores: name, config_class, translator, driver_package, dbapi_module, entrypoint
- Registry supports registration, lookup by config type, and listing all backends
- Built-in backends are registered at module load time
- Thread-safe registration (use lock)

**Tests:**
- Unit: register backend, lookup by name, lookup by config type
- Unit: concurrent registration (thread safety)
- Unit: list all backends returns built-in + registered

### REG-02: Manual Registration API

Public API for users to register custom backends.

**Acceptance:**
- `register_backend(name, config_class, translator, driver_package, dbapi_module=None, entrypoint=None)` function added to public API
- `config_class` must implement `WarehouseConfig` protocol (runtime check)
- `translator` must be callable accepting config instance, returning `dict[str, str]`
- Registering duplicate name raises `BackendAlreadyRegisteredError` with clear message
- Registration is idempotent if same config_class provided (no error)

**Tests:**
- Unit: register valid backend succeeds
- Unit: register with invalid config_class raises TypeError
- Unit: register with non-callable translator raises TypeError
- Unit: duplicate registration raises `BackendAlreadyRegisteredError`
- Unit: idempotent registration with same config_class
- Integration: manually registered backend works with `create_pool()`

### REG-03: Registry Integration

Registered backends work with existing pool creation flow.

**Acceptance:**
- `resolve_driver()` checks registry after built-in mappings
- `translate_config()` checks registry after built-in translators
- `resolve_dbapi_module()` checks registry for custom dbapi_module
- No changes to `create_pool()` signature required

**Tests:**
- Unit: `resolve_driver()` returns registered driver_package
- Unit: `translate_config()` calls registered translator
- Integration: `create_pool(registered_config)` creates working pool

### REG-04: List Backends Utility

Utility to inspect registered backends.

**Acceptance:**
- `list_backends() -> list[str]` returns all registered backend names (built-in + custom)
- `get_backend_info(name: str) -> BackendInfo` returns details about a specific backend
- `BackendInfo` dataclass includes: name, config_class, driver_package, is_builtin

**Tests:**
- Unit: `list_backends()` returns built-in backends
- Unit: `list_backends()` includes manually registered backends
- Unit: `get_backend_info()` returns correct info for built-in backend
- Unit: `get_backend_info()` raises for unknown backend

## Entry Point Discovery

### EP-01: Entry Point Group Definition

Define the entry point group contract.

**Acceptance:**
- Entry point group: `adbc_poolhouse.backends`
- Entry point name = backend name (e.g., `my_backend = my_package.backends:MyBackendConfig`)
- Entry point must point to a config class implementing `WarehouseConfig`
- Plugin packages must also provide translator via module-level `_translate_config` function or registered via decorator

**Tests:**
- Integration: dummy backend package discovered and registered automatically
- Integration: `create_pool(DummyBackendConfig(...))` works after discovery

### EP-02: Lazy Discovery

Discover entry points lazily, not at import time.

**Acceptance:**
- Entry points scanned on first `create_pool()` call (or first registry lookup)
- Scanning uses `importlib.metadata.entry_points(group="adbc_poolhouse.backends")`
- Discovered backends registered automatically
- Subsequent calls use cached registry (no re-scan)
- `force_discover()` utility to re-scan if needed (for testing)

**Tests:**
- Unit: first call triggers scan, second call uses cache
- Unit: `force_discover()` triggers re-scan
- Integration: lazy discovery works with installed dummy backend

### EP-03: Error Handling

Graceful handling of malformed or broken entry points.

**Acceptance:**
- Entry point pointing to non-existent module: log warning, skip, continue
- Entry point pointing to class that doesn't implement `WarehouseConfig`: log warning, skip, continue
- Missing translator function: log warning, skip, continue
- Import error during load: raise `BackendLoadError` with package name and original exception

**Tests:**
- Integration: `broken_backend_no_translator` logged and skipped, other backends still work
- Integration: `broken_backend_wrong_type` logged and skipped, other backends still work
- Unit: verify warning log messages contain helpful diagnostics

## Out of Scope

| Feature | Reason |
|---------|--------|
| Async plugin loading | Not needed; sync is sufficient |
| Backend unregistration | No clear use case; can be added later |
| Backend versioning/conflict resolution | Defer until real-world feedback |
| Plugin configuration validation beyond protocol | Keep minimal; translators handle backend-specific validation |
