## Architectural Decision Record

The milestone pivoted from the original 3-goal plan:

1. ~~Manual Registration~~ → Built (Phase 17), then deliberately removed (Phase 18)
2. ~~Auto-Discovery via Entry Points~~ → Invalidated (no plugin registry means no entry points)
3. **Plugin Author Documentation** → Not started (Phase 20)

To a simpler architecture:

1. **Self-describing configs** (Phase 17.5 + 18) — each config carries its own driver path, kwargs translation, dbapi module, and entrypoint
2. **Raw driver args** (Phase 19) — `create_pool(driver_path=...)` and `create_pool(dbapi_module=...)` for users who don't want config objects
3. **Protocol-based contract** (Phase 18) — `WarehouseConfig` Protocol defines the 4-method interface any third-party config must implement

Third-party configs work with `create_pool()` via the Protocol contract — no registration or discovery needed. The remaining gap is documenting this contract for plugin authors (DOC-01/02/03).
