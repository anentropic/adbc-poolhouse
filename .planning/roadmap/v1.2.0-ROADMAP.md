# Roadmap: v1.2.0

**Milestone:** Plugin/Extensibility API
**Target:** Enable 3rd party libraries to add backends by defining a config class
**Previous:** v1.1.0 — 12 backends, pytest-adbc-replay integration tests

---

## Goals

1. **Self-Describing Configs**: Config classes carry their own driver path + kwargs — no registry indirection
2. **Raw Pool Creation**: `create_pool()` overload accepting `(driver_path, db_kwargs)` directly
3. **Documentation**: Guide for plugin authors on implementing custom backends

---

All phases must pass before completion:
- Unit tests for all new public APIs
- `uv run basedpyright` passes (strict mode)
- `uv run ruff check` passes
- `uv run ruff format --check` passes

## Phase Details

### Phase 18: Registration Removal

**Goal**: Make config classes fully self-describing so the registry is unnecessary

**Depends on**: Phase 17.5

**Success Criteria** (what must be TRUE):
  1. Each config class has a method (e.g. `_driver_path()`) returning its ADBC driver path
  2. `resolve_driver()` calls the config method directly — no registry lookup
  3. `_registry.py`, `_setup_lazy_registrations()`, and all lazy registration machinery deleted
  4. `register_backend()` and `ensure_registered()` removed from public/internal API
  5. All existing tests pass (driver resolution, pool creation, semi-integration)
  6. 3rd party config classes work by defining the required methods — no registration call needed

**Plans**: TBD

---

### Phase 19: Raw create_pool Overload

**Goal**: Allow `create_pool()` to accept raw driver_path + db_kwargs instead of a config object

**Depends on**: Phase 18

**Success Criteria** (what must be TRUE):
  1. `create_pool()` accepts either a config object OR `(driver_path, db_kwargs)` via overload
  2. Type signatures are correct — basedpyright validates both call patterns
  3. Raw path avoids all config/driver resolution — goes straight to ADBC connection
  4. Existing config-based `create_pool()` calls unchanged
  5. `managed_pool()` supports the same overload
  6. Unit tests cover both call patterns

**Plans**: TBD

### Phase 20: Plugin Author Documentation

**Goal**: Document how to implement and publish custom backends

**Depends on**: Phase 19

**Success Criteria** (what must be TRUE):
  1. "Custom Backends" guide exists with step-by-step instructions
  2. Example config class included in docs (minimal working example)
  3. Required config protocol documented (to_adbc_kwargs, _driver_path, _adbc_entrypoint, etc.)
  4. `mkdocs build --strict` passes
