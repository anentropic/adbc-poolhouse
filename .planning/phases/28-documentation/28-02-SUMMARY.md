# Phase 28 Plan 02: Render Async Classes in the API Reference Summary

Render `AsyncPool`, `AsyncConnection`, and `AsyncCursor` in the API reference at their real `_async` module paths via explicit mkdocstrings blocks injected by the gen-files generator, satisfying DOCS-02 without promoting the experimental classes to `__all__`.

## What Was Built

DOCS-02 is satisfied. The built `site/reference/adbc_poolhouse/index.html` now renders:

- `AsyncPool`, `AsyncConnection`, `AsyncCursor` with their Google-style docstrings and public methods, each under an `h2` doc-heading at its real `_async._*` path.
- The three async entry-point functions (`create_async_pool`, `managed_async_pool`, `close_async_pool`) — these were already rendered by the package `::: adbc_poolhouse` block, so no extra blocks were added (Task 2 was a confirm-only task that passed on first verification).

`.venv/bin/mkdocs build --strict` exits 0. `src/adbc_poolhouse/__init__.py` is byte-for-byte unchanged — the classes stay out of `__all__` and the PEP-562 lazy `__getattr__` (D-28-02).

## How It Was Built

The three explicit blocks (each with `show_root_heading: true`, `members_order: source`, and the per-block `filters: ["!^__"]` override) plus an `## Async API (experimental)` heading are appended to the top-level package reference page.

The key implementation detail: the blocks are written by `docs/scripts/gen_ref_pages.py` (the gen-files generator that produces `reference/adbc_poolhouse.md`), NOT by hand-editing the on-disk markdown. See the deviation below for why.

The per-block `filters: ["!^__"]` REPLACES (does not merge with) the global `mkdocs.yml` filter `["!^_"]`. Without it, the global single-underscore filter would strip the `_async._*` module path and the blocks would resolve to nothing. `["!^__"]` admits the `_async._*` path while still hiding dunder members.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Blocks must be injected via the gen-files generator, not the on-disk markdown**

- **Found during:** Task 1
- **Issue:** The plan's primary approach (Task 1 `<action>`) was to append the explicit blocks to the on-disk `docs/src/reference/adbc_poolhouse.md`. That file is GENERATED by `docs/scripts/gen_ref_pages.py` via `mkdocs_gen_files.open("reference/adbc_poolhouse.md", "w")`, which writes a virtual-filesystem copy that SHADOWS and overrides the on-disk file at build time. Hand-edits to the on-disk file are silently dropped — the first build attempt rendered NONE of the three blocks (verified: no `doc-heading` for any of the three classes; the page ended mid-way inside the `managed_async_pool` source block, and the `## Async API` heading was absent from the built HTML). The on-disk `docs/src/reference/adbc_poolhouse.md` is also gitignored, confirming it is a generated artifact, not a source file.
- **Fix:** Moved the three explicit blocks (and the heading) into `gen_ref_pages.py` as a module-level `_ASYNC_REFERENCE_BLOCK` constant, appended to the package page immediately after the `::: adbc_poolhouse` line when the generator writes it. This is generation-aware and survives the virtual-FS regeneration. Restored the stray on-disk `docs/src/reference/adbc_poolhouse.md` to its canonical generated content (`# \`adbc_poolhouse\`\n\n::: adbc_poolhouse\n`) so it carries no misleading stale edits.
- **Relation to plan FALLBACK:** The plan's FALLBACK clause was triggered conceptually (the in-place on-disk approach "cannot resolve under strict build" — here, it cannot even reach the build). Rather than the plan's suggested fallback (a separate `docs/src/reference/async.md` page wired into the nav), I kept the blocks ON the package page exactly as the primary approach intended, by relocating the WRITE into the generator. This honors the plan's stated artifact target (`docs/src/reference/adbc_poolhouse.md` carries the blocks) and `must_haves.artifacts` (the page contains `adbc_poolhouse._async`) more faithfully than a separate page would, while being robust against gen-files.
- **Files modified:** `docs/scripts/gen_ref_pages.py`
- **Commit:** 77fbd63

## Notes / Observations

- **Single-underscore members now render.** The plan MANDATED the `filters: ["!^__"]` form ("Use this single canonical form on every block"; "Do NOT use `filters: []`"). That form hides only dunders, so a few single-underscore helpers now appear in the rendered reference: `AsyncConnection._enter_offload`, `AsyncConnection._exit_offload`, `AsyncConnection._offloading`, and `AsyncCursor._adbc_cancel`. This is the documented, intended consequence of the plan-chosen filter, not a build failure. If a future docs pass wants to hide these private helpers, tightening the per-block filter to also exclude single-underscore names (e.g. `["!^_"]` plus an explicit class-path admit, or marking them with a `@private` griffe directive in the docstring) would be the follow-up — out of scope here since the plan fixed the filter form.
