<objective>
Implement the cooperative-cancellation machinery (CANCEL-01/02/03 + the EDGE-17/19 fidelity contract): a new `cancellable_offload` helper (the watcher/worker anyio task group, D-25-01/08), a new `AsyncConnection.invalidate()` shielded poison-recovery method (D-25-03), and the rewire of the six `AsyncCursor` query/fetch methods to route through `cancellable_offload` and invalidate-on-cancel (D-25-04).

Purpose: This is the load-bearing implementation. A cancelled/timed-out `execute`/`fetch_arrow_table` must abort the in-flight C call via `adbc_cancel` (once, from the loop thread, shielded), join the worker (`abandon_on_cancel=False`), invalidate the poisoned connection (shielded), and re-raise the framework cancellation — never swallow it, never poison the pool. The single subtlest piece (D-25-05) is the single-member `ExceptionGroup` unwrap on the NON-cancel path that keeps a genuine `AdbcError` bare (Phase 24 EDGE-17 contract).
Output: `_cancel.py` (new), `_connection.py` (+invalidate), `_cursor.py` (six methods rewired). Behaviour is proven by the test suites in 25-03/25-04; this plan's own verification is import/static/scan-level plus existing-suite regression.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| loop thread → worker thread | The watcher (loop thread) fires `adbc_cancel()` to interrupt the worker thread's blocked C call. This is the one documented thread-safe ADBC carve-out (RESEARCH Grounding); all other cross-thread driver access is forbidden and not introduced here. |
| cancelled op → pool | A cancelled, poisoned connection must never return to the pool checked-out. The invalidate path is the boundary that drops it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-02-DOS1 | Denial of Service | pool checkout accounting under a cancel storm | mitigate | A cancelled in-flight op invalidates the connection via shielded `fairy.invalidate()`; `pool.checkedout()` returns to 0 (CANCEL-02). Asserted by 25-03/25-04 EDGE-02/05 and the real-driver leg; the shield guarantees cleanup completes even if a second cancel arrives (D-25-07) |
| T-25-02-DOS2 | Denial of Service | stranded non-cancellable worker hanging the loop | mitigate | `abandon_on_cancel=False` joins the worker after `adbc_cancel` unblocks it; the watcher never abandons. 25-03/04 prove no hang via `real_clock_watchdog` + the ×20 loop |
| T-25-02-TMP | Tampering / Information | reuse of a poisoned (aborted-transaction) connection serving stale data | mitigate | Mandatory invalidate on the cancel path (probe-confirmed poisoning, D-25-03); never sniff-to-recover; never return-busy |
| T-25-02-ID | Information Disclosure | cancel path leaking a spurious "Interrupted!" driver error to the caller | accept→mitigate | anyio's cancel collapse swallows the worker interrupt on the cancel path automatically; the wrapper does NOT message-sniff (D-25-02). The caller sees only the framework cancellation (`TimeoutError`/none) |
| T-25-02-SC | Tampering | npm/pip/cargo installs | accept | No package installs; no new third-party dependency (RESEARCH Package Legitimacy Audit: none added) |
