<objective>
Land the Wave-0 harness prerequisite the rest of Phase 24 rides on: make the
BlockingStubCursor gate **re-armable per blocking call** (so one cursor can block on
`execute` then block again on `fetch_arrow_table`) AND make the loop-facing `entered`
signal fire **after** the worker is inside the blocked section (D-CF-01), landing both
together. A naive re-arm without the timing fix deadlocked ~33% of the time in Phase 23
(commit reverted; see `23-REVIEW-FIX.md` WR-01) — this plan fixes the root cause, not the
symptom.

Purpose: every execute-then-fetch EDGE test in Plan 04 (EDGE-09/12/15/26) needs a cursor
that re-blocks on its second call and a `entered` that is a true "inside execute" signal.
Without this, those tests either cannot be written or deadlock under full-suite scheduling.

Output: a verified-green, loop-stable harness change in `tests/_async_harness/` — no
`src/` changes. This is purely test infrastructure (TEST-05 carry-forward).
</objective>
