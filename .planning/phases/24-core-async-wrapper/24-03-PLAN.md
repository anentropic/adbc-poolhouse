<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| task ↔ task (aliasing) | two tasks may capture the same AsyncConnection/AsyncCursor; the `_in_use` flag is the boundary guard |
| event loop ↔ worker thread | every DBAPI call crosses here via the offload; the parent's `_in_use` flag and the limiter bound this |
| connection lifetime ↔ checkin | a checked-out connection must always be returned/invalidated; the shield guards cancel-during-cleanup |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-03-EOP | Elevation of Privilege | concurrent aliasing lets two workers run C calls on one connection (interleaved statements in one txn) | mitigate | `_in_use` check-and-set with no intervening await raises `ConnectionBusyError` on the 2nd entry (D-24-03, Task 1); EDGE-15 proves max_concurrent==1 (Plan 04) |
| T-24-03-DOS | Denial of Service | connection leaked from the pool if checkin is abandoned on cancel, or if `__aenter__` setup raises | mitigate | shielded checkin (`CancelScope(shield=True)`) + reclaim-on-failure in `__aexit__` (EDGE-18, Task 1) |
| T-24-03-INFO | Information Disclosure | ADBC error type/traceback altered crossing the thread boundary | mitigate | no re-wrap; `run_sync` re-raises verbatim (EDGE-17/ACUR-06, Task 2) |
| T-24-03-LIFE | Information Disclosure | use-after-checkin: a live Arrow reader bound to a closed cursor returns freed memory | mitigate | `fetch_arrow_table` returns a self-owning materialized `pyarrow.Table`, not a reader (EDGE-21, Task 2; Pitfall 7) |
| T-24-03-RACE | Tampering | the `_in_use` check raced by an injected `await` between check and set | mitigate | enforce single synchronous check-and-set span; reviewed in source + proven by EDGE-15 |
| T-24-03-SC | Tampering | npm/pip/cargo installs | N/A | no new packages |
</threat_model>
