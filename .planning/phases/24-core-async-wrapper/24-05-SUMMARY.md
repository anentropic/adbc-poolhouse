# Phase 24 Plan 05: Async docs gate Summary

The per-phase documentation completion gate (CLAUDE.md, phase >= 7) for the async
surface: an honest async usage guide, the forbidden connection-aliasing antipattern
documented with `ConnectionBusyError`, complete Google-style docstrings with
`Example:` blocks on every public async symbol, and a green `mkdocs build --strict`.

## What Was Built

### Task 1 — AsyncPool Example block (commit `d664118`)

The four `_async/` modules already carried complete Google-style Markdown docstrings
(Args/Returns/Raises) from Plans 02/03, with `Example:` blocks on `create_async_pool`,
`managed_async_pool`, `close_async_pool`, `AsyncConnection`, and `AsyncCursor`. The
one gap was `AsyncPool`, which had no `Example:`. Added an `Example:` admonition
(singular, fenced `python`) showing the full
`create_async_pool → connect → cursor → execute → fetch_arrow_table → close` flow,
bringing the `_async/` `Example:` count to 6.

Verified the rest of the Task 1 contract was already satisfied: no RST role syntax
(`:func:`/`:class:`/`:meth:`) anywhere in `_async/`; `ConnectionBusyError`'s docstring
(in `_exceptions.py`) states the serialized-not-concurrent rule; `cursor()` is
documented as synchronous (ACONN-03); the sync property passthroughs are documented
(ACUR-07); and `fetch_arrow_table` is documented as returning a materialized
`pyarrow.Table` safe after check-in (ACUR-04). No `docs/src/reference/` files were
hand-written — that tree is auto-generated by `gen_ref_pages.py` from these docstrings.

### Task 2 — async usage guide + nav/index (commit `80a94f0`)

- `docs/src/guides/async.md`: a how-to guide in the project voice (second person,
  direct/practical). It walks the canonical flow `create_async_pool` →
  `await pool.connect()` → `conn.cursor()` (sync) → `await cur.execute(...)` →
  `await cur.fetch_arrow_table()` → check-in via `async with`, and calls out the two
  deliberate non-coroutines (`cursor()` and the sync property reads).
- **Honest concurrency section.** Quotes the Phase 22 SPIKE go/no-go directly:
  `execute` (and other I/O-bound C calls) parallelize because ADBC releases the GIL
  (~2.77x@N=4, ~69% efficiency), but `fetch_arrow_table` materialization partially
  serializes on the GIL (~1.67x@N=4, ~42% efficiency). Notes the numbers came from an
  in-process DuckDB driver (so they capture GIL behaviour, not network concurrency)
  and that a networked backend has real I/O latency to overlap. States the per-pool
  `CapacityLimiter(pool_size + max_overflow)` bound (CORE-02). This is the DOCS-01
  honesty framing, delivered early (DOCS-01 formally lands in Phase 28).
- **Forbidden-aliasing antipattern.** The canonical wrong (aliased cursor across a
  task group → `ConnectionBusyError`) / right (one connection per task) example from
  CONTEXT D-24-03, plus the "why a hard error, not silent queuing" rationale (silent
  serialization still interleaves statements inside one transaction — driver-safe,
  logically corrupt — and hides the bug).
- **Shielded cleanup section.** Documents that check-in and `close_async_pool` run a
  shielded teardown, and that full in-flight query cancellation lands in a later
  release (Phase 25) without over-claiming it ships now.
- Notes the `[async]` extra install, referencing it as the install path (formally
  declared in Phase 26) without over-claiming.
- `mkdocs.yml`: an "Async Pool" nav entry under Guides (after Pool Lifecycle).
- `docs/src/index.md`: an "Async" section with a runnable DuckDB example and a
  what's-next link to the guide.
- Humanizer pass applied to all new prose.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] method-level mkdocstrings cross-ref broke `mkdocs --strict`**
- **Found during:** Task 2 (first strict build).
- **Issue:** The guide initially linked
  `[`fetch_arrow_table`][adbc_poolhouse._async._cursor.AsyncCursor.fetch_arrow_table]`.
  `AsyncCursor` is not a top-level package export and the deep-module method path is
  not a registered autorefs anchor, so `mkdocs_autorefs` emitted "Could not find
  cross-reference target", which aborts under `--strict`.
- **Fix:** Replaced that single cross-ref with inline code (`` `fetch_arrow_table` ``).
  The other cross-refs in the guide point at top-level exports
  (`adbc_poolhouse.create_async_pool`, `managed_async_pool`, `close_async_pool`,
  `ConnectionBusyError`), which resolve cleanly.
- **Files modified:** docs/src/guides/async.md
- **Commit:** `80a94f0`

### Interpretation notes (not code changes)

- **Docstrings were not rewritten.** The Task 1 action says "audit and ensure complete
  docstrings"; Plans 02/03 had already delivered them to spec. The only change Task 1
  needed was the missing `AsyncPool` `Example:` block. The audit confirmed the rest
  (no RST, sync-cursor/property/materialization documentation, `ConnectionBusyError`
  rule) was already in place.
- **`../reference/` relative links are INFO, not strict failures.** `mkdocs` logs
  "unrecognized relative link 'reference/'" at INFO level for `index.md` and the new
  guide, exactly as it already does for the pre-existing `index.md` and
  `configuration.md` links. These do not abort `--strict` (confirmed rc=0).
- **mkdocs-material 2.0 advert.** The build prints a vendor notice recommending
  "ProperDocs"; it is a plugin advert, not a warning that fails strict mode. Silenced
  in the verification runs with `DISABLE_MKDOCS_2_WARNING=true` for readable output;
  the build passes with or without it.
