<objective>
Build the runtime harness machinery: the pure-threading `BlockingStubCursor`/`BlockingStubConnection` fakes (D-03/D-04), the `run_blocking` offload glue that bridges the `entered` signal worker→loop (Pattern 3), and the `virtual_clock` backend-dispatching façade (D-01). These three modules are the arrange/trigger surface every later EDGE test rides on; their public attribute names are a HARD CONTRACT consumed by Phases 24/25/27.

Purpose: Event-gating is the PRIMARY determinism mechanism (D-02) — the forever-blocking stub released by the test or by `adbc_cancel` is what lets cancel/timeout tests run with zero wall-clock sleeps. The virtual clock covers the deadline/timeout paths. The `entered` bridge eliminates the race where a trigger lands before the worker is actually inside `execute` (Pitfall 2). NOTE the deliberate naming overlap that must NOT be conflated: the stub exposes its own `entered` (a `threading.Event` set inside `_block()`) AND `run_blocking(...)` takes a SEPARATE `entered` (an `anyio.Event` set via `from_thread.run_sync`). Same name, different objects, different uses — both modules must document the distinction so a Phase 24 author never `await stub.entered.wait()`s a threading.Event on the loop (which would reintroduce Pitfall 2).
Output: `tests/_async_harness/stubs.py`, `gating.py`, `clock.py`. Async dual-backend self-tests for these modules land in Plan 04 at `tests/_async_harness/test_harness.py` (they require the full anyio_backend machinery, which is only visible to test files at/below the conftest's package); this plan's verification is import/typecheck-level. The guard (Plan 03) is independent and runs in parallel.
</objective>

| Threat ID | Category | Component | Disposition | Mitigation Plan |
