## Findings (ranked)

### 1. Cancel path can surface a multi-member `ExceptionGroup` when `on_abort` fails — Medium (correctness)
**File**: `src/adbc_poolhouse/_async/_cancel.py:218-220`

`cancelled_by_us` is set `True` only *after* both `adbc_cancel()` and `await on_abort()`
return cleanly (lines 176-183). On a real cancel, `adbc_cancel()` interrupts the worker,
which then raises the driver's interrupt error. If `on_abort` (the connection's
`invalidate`) *also* raises, the task group exits with a **two-member**
`BaseExceptionGroup` `[invalidate_error, driver_interrupt]`, `cancelled_by_us` stays
`False`, so the `len(eg.exceptions) == 1` unwrap at line 218 is skipped and line 220
re-raises the **group verbatim**.

The docstring (line 217) promises a failed `on_abort` "also reaches here, bare" — true
only for the single-member case. When `adbc_cancel` succeeds *and* `invalidate` fails, the
caller's cancel scope receives a `BaseExceptionGroup` instead of a cancellation or a bare
error, breaking structured-cancellation expectations; the connection may also be left
partially-invalidated.

**Failure scenario**: `with fail_after(t): await cur.execute(sql)` is cancelled mid-call;
`adbc_cancel` fires (worker raises the driver interrupt); `fairy.invalidate()` raises (e.g.
the underlying dbapi connection is already broken). Caller sees `BaseExceptionGroup`, not
`CancelledError`.

**Suggestion**: On the non-cancel branch, when the group carries the expected driver
interrupt plus an `on_abort` failure, raise the `on_abort` failure bare (or re-raise the
cancellation with the `on_abort` failure chained). Reachable only when `invalidate` itself
fails, so narrow — but it is the one path that silently changes the surfaced exception type.

### 2. `commit()` / `rollback()` are not cancellable — Medium-Low (robustness)
**File**: `src/adbc_poolhouse/_async/_connection.py:178`, `:195`

Cursor methods route through `cancellable_offload` (wired to `adbc_cancel` + poison
recovery). `commit()`/`rollback()` use the plain `offload` with `abandon_on_cancel=False`
and no `adbc_cancel` hook. A timeout/cancellation around a blocking commit therefore cannot
interrupt it — the loop defers the cancellation and waits to join the worker until the
driver call returns on its own.

**Failure scenario**: `with fail_after(5): await conn.commit()` against a slow/stalled
server — the 5s deadline does not abort the commit; the call blocks until the driver
returns, then the cancellation surfaces.

**Suggestion**: Confirm this is intended. The cursor docstrings document the cancel path
explicitly; the txn methods are silent about it. Either wire `cancellable_offload`
(+ poison recovery) for commit/rollback, or document that they are deliberately
uninterruptible so callers don't assume a timeout aborts them.

### 3. `__getattr__` mislabels *any* async-import failure as "install [async]" — Low-Medium (correctness/DX)
**File**: `src/adbc_poolhouse/__init__.py:88-96`

The lazy loader catches **all** `ImportError` from `import _async` and re-raises
"requires the optional async dependencies … pip install adbc-poolhouse[async]". If `anyio`
is installed but the async subpackage fails to import for any *other* reason (a bug in
`_async`, a broken transitive import, a partially-installed env), the user is told to
install an extra they already have — sending them down the wrong path and masking the real
`ImportError`.

**Suggestion**: Narrow the catch — only relabel when the missing module is actually `anyio`
(e.g. check `exc.name == "anyio"`, or `isinstance(exc, ModuleNotFoundError)` naming anyio);
re-raise everything else unchanged.

### 4. Offload guard `try/finally` duplicated ~10× — Low (cleanup / maintainability)
**Files**: `src/adbc_poolhouse/_async/_cursor.py` (execute, executemany, fetchone,
fetchmany, fetchall, fetch_arrow_table, close) and `_connection.py` (commit, rollback,
close)

The `_owner._enter_offload(); try: … finally: _owner._exit_offload()` (or
`self._enter/_exit`) bracket is copy-pasted across ~10 methods. A single
`@contextlib.asynccontextmanager` helper (`async with self._owner._offloading():`) would
collapse the duplication and make it impossible to forget the `finally`.

---

### 5. `offload` allocates a new `CapacityLimiter(math.inf)` per call — Low (efficiency)
**File**: `src/adbc_poolhouse/_async/_offload.py:104`

Every offloaded call constructs `inner_limiter = anyio.CapacityLimiter(math.inf)`. Real
concurrency is already bounded by the per-pool token held across `async with limiter`, so
the inner limiter is a constant. A module-level `_UNBOUNDED = anyio.CapacityLimiter(math.inf)`
reused across calls avoids the per-call allocation on the hot path.

---

### 6. `fetchmany` duplicates the whole `cancellable_offload` call for the size arg — Low (cleanup)
**File**: `src/adbc_poolhouse/_async/_cursor.py:293-306`

The `if size is None / else` branches differ only by passing `size`. Normalising the args
once (`args = () if size is None else (size,)`) and making a single `cancellable_offload(…,
*args, …)` call removes the near-identical duplication and the risk of the two paths
drifting.
